## The Actual Question

Underneath all the technical architecture — the markdown files, the git history, the publishing pipeline, the graph view — there is a question that none of the tools can fully answer:

**What does it mean to build something that outlasts you?**

Not physically. Not in the sense of legacy or reputation. In the literal, practical sense: what would it take for the knowledge you've accumulated to remain useful to someone else, after you're no longer there to explain it?

This is a harder question than it sounds. Most knowledge is tacit. Most of what makes an archive useful is the context that surrounds it — the sense of why things matter, how they connect, what questions they're answering. That context lives in people. When the people leave, the context leaks out.

A knowledge system built for continuity is trying to externalize that context. To make the relationships explicit. To build a structure that carries meaning forward even when the person who built it isn't there to carry it.

---

## What This Research Is Trying to Prove

The hypothesis is simple: a well-designed, low-cost, markdown-based knowledge architecture can preserve significantly more institutional knowledge than the current default — which is, in most small organizations, no architecture at all.

Not because the tools are magic. Because structure is durable in a way that informal practices aren't.

The files you're reading right now are plain text. They will be readable in any environment, on any machine, with any software, for as long as plain text is a thing. The links between them encode relationships that survive the transition from one tool to another. The git history records every change. The publishing layer makes the whole thing navigable to anyone with a browser.

This is what remains when you build with intention.
